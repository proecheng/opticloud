## 1. 定位

> **⚠️ 重要前提说明**（v0.5.1 补充）
>
> 1. **当前网站没有本地大模型**。v1 全部 Chat / NL→Model 流量走 **DeepSeek API**（外部商业 API，按 Token 付费）。
> 2. **自研算法 License 已确定 Apache 2.0**（课题组接受），后续合作分润、外部接入、社区贡献均按此基准。
> 3. **本地大模型纳入 v3**，需要 GPU 集群（A100×8 或同等）一次性投入 ¥80-150 万 或月租 ¥3-8 万；触发条件为月活付费用户 ≥ 500。详见 **§22 分阶段预算**。
> 4. **预算控制原则**：先做小、跑通商业闭环、再扩本地大模型 / 自建 GPU / 私有部署等重资产。

一句话：**"描述你的问题，我们替你建模、求解、预测、解释"** —— 面向通用行业的 **AI-Native 决策与预测云**。

- **入口通用化**：用户既可以直接调用 SKU（API / 控制台），也可以用一段中文/英文自然语言描述需求，由 LLM 智能层完成建模、求解器选型、参数填充。
- **算法多源**：底座包含课题组自研算法（**已通过代码自查的**，详见 §4.5）、开源算法（HiGHS / OR-Tools / Chronos 等）、以及通过 **Provider Registry** 接入的第三方 API。
- **输出多模态**（v1 仅 JSON + 仪表盘 + NL 解读三件套；其它 v2+）。
- **首发语种**：中文为主，关键页面（Landing / Pricing / API 错误码 / 法律页）配英文兜底；全站英文 v1.5；详见 §15。
- **商业模式**：订阅 + 按量（Credits），参考 [openclaudecode.cn/console](https://www.openclaudecode.cn/console)。

## 3. 核心产品形态：四种入口（v1 只做 A+B；C/D 后置 v2）

### 入口 A：Chat 聊天界面 — **v1 主入口**（仿 ChatGPT 形态）

```
┌─────────────────────────────────────────────────────────────┐
│ 历史会话 (左侧)        │   主对话区                            │
│  ─ 上海冷链 VRP 7/8     │  🤖 你好，描述一下你要解决的问题…    │
│  ─ 光伏出力预测 7/5     │  👤 我有20台车，5个仓库，80个客户…   │
│  ─ 投资组合优化 6/30    │  🤖 已识别为 VRPTW；决策变量/目标/   │
│                         │       约束如下…                      │
│ 模型选择（v1 默认）：   │       [运行求解器] [修改模型] [转API]│
│  · DeepSeek-V3.5 ✅     │  📎 上传数据   💎 预估消耗 ≤ 12 积分│
│ + 新对话                 │                                      │
└─────────────────────────────────────────────────────────────┘
```

**关键能力**（v1 全部具备）：

| 能力 | 说明 |
|---|---|
| **多轮对话** | 上下文累积保留 |
| **流式输出** | Token 级流式（SSE / WebSocket） |
| **文件附件** | CSV / Excel / JSON / 时序数据 |
| **建模卡片** | 结构化展示决策变量 / 目标 / 约束；术语悬停气泡 |
| **一键求解** | 卡片下方 [运行求解器]，调用底层 |
| **求解中可视化** | 长任务实时迭代曲线 |
| **结果回流** | 图表 + NL 解读贴回对话 |
| **What-If 追问** | 改约束自动重跑 |
| **可转 API** | 一键生成 cURL / Python SDK 代码 |
| **会话分享** | 只读链接，可加密 |

**v1 不做 / 后置 v2**：
- 多 LLM 切换（v1 只接 DeepSeek API，详见 §8.3）
- 自托管本地 LLM（成本太高，v3+）
- 会话语义检索（依赖 Vector DB，v2 接入后）

**LLM 智能层后端**（仿 OptiMUS-0.3 / OptimAI 的 4-Agent 范式）：

```
用户消息 → Router LLM（判断闲聊/建模/追问/数据/求解触发）
              ├─ 闲聊 → 直接回复
              ├─ 建模 → Formulator → Planner → Coder ──[沙箱执行]──→ Critic
              ├─ 数据 → Data Profiler
              └─ 求解 → Provider Registry → Result Renderer → 流式回流
```

> **Coder 输出的代码必须在沙箱（gVisor / Firecracker）内运行**，禁止访问外网与宿主文件系统，详见 §7。

### 入口 B：SKU 直调（API / SDK）— **v1 必上**

```http
POST /v1/optimize/vrptw
Authorization: Bearer sk-xxx
Content-Type: application/json

{ "depots": [...], "vehicles": [...], "customers": [...], "max_solve_seconds": 60 }
```

### 入口 C：控制台无代码工作流 — **v2 推出**

类 Nextmv / RapidMiner 的拖拽编辑器；v1 不做。

### 入口 D：新手模式（行业模板向导）— **v1.5（M5）推出**

参见 v0.4 详细设计；v1 暂以 Chat 内"问题类型卡片"模拟简化版。

## 5. 与外部 API 的对接 — Provider Registry

### 5.1 市场调研结论

把 **NL→Model + 多源 Provider + 订阅市场** 三件事合一的产品，国内基本空白，海外只有 Nextmv 接近一半（缺 NL 入口和订阅市场）。

| 类别 | 代表产品 | 与本站的差异 |
| --- | --- | --- |
| 求解器云 | Gurobi Cloud、AMPL、Frontline | 需用户写数学模型；无 NL 入口 |
| DecisionOps | [Nextmv](https://www.nextmv.io/) | 开发者基础设施，非 SKU 化订阅 |
| 预测基础模型 | [Nixtla TimeGPT](https://www.nixtla.io/)、Amazon Forecast | 只预测、无优化、无 NL |
| AutoML | [DataRobot](https://docs.datarobot.com/en/docs/api/reference/predapi/index.html)、RapidMiner | 优化弱、私有部署贵 |
| NL→Model 学术 | [OptiMUS](https://arxiv.org/abs/2310.06116) / [OptimAI](https://arxiv.org/abs/2504.16918) / [LEAN-LLM-OPT](https://arxiv.org/pdf/2601.09635) | 学术原型，无中文 SaaS |
| API 市场 | [RapidAPI](https://rapidapi.com/) | 通用聚合，非优化/预测垂直 |

### 5.2 Provider Registry

```
Provider Registry
   ├── 自研 Runner（Python 模块）
   ├── ITADN AIGC 运行时（Docker）
   ├── 课题组成员 A 的 HTTP API
   ├── 开源 Runner（HiGHS / OR-Tools / Chronos / …）
   └── 商业求解器 Gurobi Cloud（v2，用户自带 license）
```

统一接口：

```python
class AlgorithmProvider(Protocol):
    capabilities: list[Capability]
    def health(self) -> HealthStatus: ...
    def submit(self, job: JobSpec) -> JobHandle: ...
    def fetch(self, handle: JobHandle) -> JobResult: ...
    def cancel(self, handle: JobHandle) -> None: ...
    def cost_estimate(self, job: JobSpec) -> CostEstimate: ...
```

详细治理 → §18。

4 项：OpenAPI 3.0 描述 / 健康检查 / 示例 IO / 离线评测报告。课题组合作方可用我方 "Provider SDK 脚手架"。

## 8. 算法分层（v0.5 优化）

### 8.1 优化算法分层（T6 拆细 + T3 加 CP-SAT）

| 层级 | 名称 | 数学结构 | v1 默认后端 | k_opt |
| :--: | --- | --- | --- | :--: |
| **T1** | LP / 凸 QP | LP, QP（凸） | HiGHS / OSQP | **1.0×** |
| **T2** | 凸 NLP / SOCP / SDP | Convex NLP, SOCP, SDP | CVXPY + SCS | **2.0×** |
| **T3a** | MILP / IP | 混合整数线性 | HiGHS MIP / CBC | **4.0×** |
| **T3b** | **CP-SAT / 约束规划** | 约束 + SAT 混合 | OR-Tools CP-SAT | **4.0×** |
| **T4** | 非凸 NLP / 全局 / 黑盒 / 贝叶斯 | Nonconvex NLP | IPOPT + multi-start / CMA-ES / Optuna；**AQGS（自研）** | **6.0×** |
| **T5** | MINLP（凸/非凸） | 离散 + 非线性 | Bonmin / Couenne；**Trust-Tech MINLP（自研）** | **10.0×** |
| **T6a** | 多目标 / 帕累托 | Multi-objective | pymoo（NSGA-II/III） | **8.0×** |
| **T6b** | 随机规划 | Stochastic Programming（场景法/SAA） | Pyomo + 自研场景生成 | **12.0×** |
| **T6c** | 鲁棒 / DRO | Robust / Distributionally Robust | ROmodel + 自研 | **15.0×** |
| **T6d** | 双层 / 博弈 | Bilevel / Game / Stackelberg | 自研 + bilevel-jump | **18.0×** |

> **元启发式**（GA/PSO/SA/CMA-ES）作为 T4 / T5 备选 Provider，价格系数与所在层一致。
>
> **组合优化模板**（VRP/TSP/Bin Packing/Scheduling）按底层公式归 T3a 或 T3b。

### 8.2 预测算法分层

| 层级 | 名称 | v1 默认后端 | k_pred |
| :--: | --- | --- | :--: |
| **P1** | 经典统计 | Nixtla statsforecast、statsmodels、Prophet | **1.0×** |
| **P2** | 经典 ML（CPU） | scikit-learn / XGBoost / LightGBM / CatBoost / PyOD；**TT-KMeans（自研）** | **1.5×** |
| **P3** | 浅层 DL（GPU） | PyTorch + Nixtla neuralforecast / Darts；**用 CPSOTJUTT 训练的具体网络**作为 SKU | **4.0×** |
| **P4** | 大 DL（GPU 重） | HuggingFace Transformers / GluonTS / PyG | **8.0×** |

> **置信带 / 不确定性量化**：每层 +0.5×。
>
> **AutoML**（AutoGluon / FLAML）为元层，**按目标 SKU 的层级计费**，再 +20% 元层费用。

## 9. 点数（Credits）计费体系（v0.5 修正）

### 9.1 为什么选点数制

抽象化、预付防超支、国际化、促销灵活、算法分层友好。参考 [Data-Mania](https://www.data-mania.com/blog/ai-pricing-models-explained-usage-seats-credits-outcome-based-options/)、[Metronome](https://metronome.com/blog/2026-trends-from-cataloging-50-ai-pricing-models)。

### 9.2 点数单位

> **1 ¥ = 100 Credits**（国内严格等价，财务可直接对账）
> **$1 = 700 Credits**（海外）

### 9.3 计费公式 — **改为封顶计费**

```
预扣（封顶）：
   Credits_max = ⌈ k_algo × scale × t_solve_cap + k_overhead ⌉ + NL_credits

实际扣（事后）：
   Credits_real = ⌈ k_algo × scale × t_solve_actual + k_overhead ⌉ + NL_credits
                ≤ Credits_max
```

| 符号 | 含义 |
| --- | --- |
| `k_algo` | T1=1.0 / … / T6d=18.0；P1=1.0 / … / P5=12.0 |
| `scale` | 问题规模归一化（每个 SKU 文档明确口径，如 VRPTW = 客户数 / 100；ACOPF = 节点数 / 1000）|
| `t_solve_cap` | 用户在请求中指定的 `max_solve_seconds`，超时强制停止 |
| `t_solve_actual` | 实际秒数 |
| `k_overhead` | 固定 5 Credits |
| `NL_credits` | `tokens / 1000 × N_x 倍率`（见 §9.6.1）|

**用户体验**：前端只显示 "💎 最多消耗 605 Credits，余额 9,200"，确认才下单；事后多退少补。

### 9.6 Provider 分润（v0.5 分级）

```
本次调用总 Credits → 拆分：
  ① overhead 5 Credits → 100% 归平台
  ② NL Token Credits  → 100% 归平台
  ③ 算法核心 Credits  → 按 Provider 类型分润：
       · 自研 Provider         → 100% 平台 / 0% 外部
       · 合作课题组 Provider   → 40% 平台 / 60% Provider
       · 商业 Provider（透传） → 50% 平台 / 50% Provider（含透传成本）
       · 开源 Runner（无 Provider 主体）→ 100% 平台
```

**算例**：VRPTW 总 605 Credits（¥6.05）
- overhead 5 + NL 50 + 算法核心 550
- 自研：平台 605
- 合作课题组：平台 5 + 50 + 550×40% = 275 / Provider 330
- 商业 Provider：平台 5 + 50 + 550×50% = 330 / Provider 275
